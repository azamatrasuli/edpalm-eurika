## Архитектура RAG

1. **Loader** (`app/rag/loader.py`): markdown → sections → chunks (max 1500 chars, 150 overlap) → embeddings → pgvector
2. **Search** (`app/rag/search.py`): query → embed → cosine similarity → top_k=5, threshold=0.3
3. **Agent** вызывает `search_knowledge_base(query)` → получает chunks → формирует ответ

## Как создать тестовый раннер

Создай `test_runner.py` в backend/:
1. Генерация external-токенов
2. Multi-turn диалоги через SSE
3. **Дополнительно:** для каждого сценария сохраняй какие tools_called были (чтобы убедиться что search_knowledge_base вызван)
4. Возврат полного текста ответа для сравнения с KB

## Методология проверки

Для КАЖДОГО сценария:
1. Запусти тест через test_runner.py
2. Проверь: вызван ли `search_knowledge_base`
3. Открой соответствующий KB файл и СРАВНИ ответ с реальными данными
4. Отметь: EXACT MATCH / CORRECT BUT INCOMPLETE / WRONG / HALLUCINATION / NOT IN KB

## Критерии качества

- **0 галлюцинаций** в ценах, датах, именах, процентах
- `search_knowledge_base` вызывается на ВСЕ 100 сценариев
- Если данных нет в KB → агент честно говорит "уточню у менеджера"
- Если данные есть → ответ ТОЧНО совпадает с KB
- Не придумывает конкретные числа (кол-во уроков, проценты) если их нет в KB

## Инструкции

1. **Прочитай ВСЕ KB файлы** перед тестированием (8 sales + 1 support)
2. Создай test_runner.py
3. Субагенты по 5, пауза 5с
4. Каждый ответ СВЕРЯЙ с KB (это главное!)
5. Галлюцинация = CRITICAL BUG → фикс промпта
6. Пробел в KB = заметка для заказчика
7. Итоговая таблица: точность по блокам
